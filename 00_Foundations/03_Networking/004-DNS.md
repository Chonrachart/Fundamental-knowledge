# DNS (Domain Name System)

- DNS is a name-to-address translation system that maps human-readable domain names to IP addresses.
- It works through a hierarchical resolution chain: browser cache, OS cache, recursive resolver, root server, TLD server, authoritative server.
- DNS enables transparent IP changes, geographic load balancing, and latency-based routing without affecting end users.

# Architecture

```text
+-----------+      +----------------+      +-------------+
|  Client   | ---> |   Recursive    | ---> | Root Server |
| (Browser) |      |   Resolver     |      | (13 clusters)|
+-----------+      +----------------+      +-------------+
                          |                       |
                          v                       v
                   +-------------+        +-------------+
                   | Authoritative| <---- | TLD Server  |
                   |   Server    |        | (.com .org) |
                   +-------------+        +-------------+
                          |
                          v
                   Returns IP to client
```

# Mental Model

```text
User types www.example.com
  |
  v
1. Browser cache hit? -----> yes --> use cached IP
  |  no
  v
2. OS cache hit? ----------> yes --> use cached IP
  |  no                       (/etc/hosts, systemd-resolved)
  v
3. Recursive resolver
  |
  v
4. Root server -----------> "ask .com TLD server"
  |
  v
5. TLD server (.com) -----> "ask ns1.example.com"
  |
  v
6. Authoritative server --> returns 93.184.216.34
  |
  v
7. IP cached (per TTL), returned to browser
  |
  v
8. Browser connects to 93.184.216.34
```

Example lookup:

```bash
dig www.example.com

# ;; ANSWER SECTION:
# www.example.com.    3600    IN    A    93.184.216.34
```

# Core Building Blocks

### Domain Name Structure

| Sub-domain | Domain | Top-Level Domain (TLD) |
| :--------: | :----: | :--------------------: |
|    www     | google |          com           |

- Domain names are hierarchical, read right to left (root -> TLD -> domain -> subdomain).
- TLD examples: `.com`, `.org`, `.net`, country codes like `.uk`, `.jp`.

Related notes: [005-http-https](./005-http-https.md)

### DNS Components

- **Domain Name** -- human-readable address (e.g., `example.com`)
- **IP Address** -- numeric address the domain resolves to
- **DNS Resolver** -- server that performs lookups on behalf of clients (e.g., `8.8.8.8`)
- **Root Servers** -- 13 logical clusters; direct queries to TLD servers
- **TLD Servers** -- manage top-level domains; point to authoritative servers
- **Authoritative Servers** -- hold the actual DNS records for a zone

Related notes: [003-addressing-and-routing](./003-addressing-and-routing.md)

### DNS Record Types

These are entries stored in a DNS zone.

- **A** -- maps domain to IPv4 address: `example.com A 10.100.100.106`
- **AAAA** -- maps domain to IPv6 address: `example.com AAAA 2606:2800:220:1:248:1893:25c8:1946`
- **CNAME** -- alias record: `www.example.com CNAME example.com` (www points to another domain)
- **MX** -- mail server for domain: `example.com MX mail.example.com` (used by email systems)
- **NS** -- nameservers for domain: `example.com NS ns1.example.com`
- **PTR** -- reverse DNS lookup: `34.216.184.93 PTR example.com`
- **TXT** -- stores arbitrary text (SPF, DKIM, domain verification)

Related notes: [005-http-https](./005-http-https.md)

### DNS Types

- **Forward DNS** -- domain to IP resolution (standard lookup)
- **Reverse DNS** -- IP to domain resolution (uses PTR records, queries `in-addr.arpa` zone)

Related notes: [003-addressing-and-routing](./003-addressing-and-routing.md)

### TTL (Time To Live)

- DNS records include a TTL value (in seconds) that controls how long results are cached.
- Lower TTL = faster propagation of changes, higher query load.
- Higher TTL = reduced query load, slower propagation.
- Example: `example.com A 10.100.100.106 TTL=3600` (cached for 1 hour)

Related notes: [005-http-https](./005-http-https.md)

# Practical Command Set (Core)

```bash
# Basic DNS lookup
dig example.com

# Query specific record type
dig example.com MX
dig example.com AAAA
dig example.com NS

# Short answer only
dig +short example.com

# Query a specific DNS server
dig @8.8.8.8 example.com

# Reverse DNS lookup
dig -x 93.184.216.34

# Trace full resolution path
dig +trace example.com

# nslookup (simpler alternative)
nslookup example.com
nslookup -type=MX example.com

# Check local DNS cache (systemd)
resolvectl query example.com
resolvectl statistics
```

# Troubleshooting Guide

```text
DNS not resolving?
  |
  +--> Check /etc/resolv.conf -- correct nameserver?
  |       |
  |       +--> dig @8.8.8.8 example.com -- works with public DNS?
  |               |
  |               +--> yes --> local resolver issue (check systemd-resolved, dnsmasq)
  |               +--> no  --> check if domain exists: dig +trace example.com
  |
  +--> Resolves but wrong IP?
  |       |
  |       +--> Check TTL -- stale cache? flush with: resolvectl flush-caches
  |       +--> Check /etc/hosts -- local override?
  |
  +--> Intermittent failures?
          |
          +--> Check multiple resolvers (dig @8.8.8.8 vs dig @1.1.1.1)
          +--> Check for DNS round-robin returning different IPs
```

# Quick Facts (Revision)

- DNS maps domain names to IP addresses (and vice versa with PTR records).
- Resolution order: browser cache -> OS cache -> recursive resolver -> root -> TLD -> authoritative.
- There are 13 logical root server clusters worldwide.
- TTL controls how long DNS answers are cached; lower TTL = faster updates.
- Common record types: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), NS (nameserver), PTR (reverse), TXT (text).
- Forward DNS = domain-to-IP; reverse DNS = IP-to-domain (PTR + in-addr.arpa).
- `dig +trace` shows the full resolution path from root to authoritative.
- DNS uses UDP port 53 by default; falls back to TCP for large responses or zone transfers.
