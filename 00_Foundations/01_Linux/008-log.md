# Logging

- Logs are timestamped records of what the system and its services did — essential for debugging, auditing, and monitoring.
- Modern Linux uses two parallel systems: classic text files in `/var/log/` and the systemd binary journal (read via `journalctl`).
- Log rotation (`logrotate`) prevents logs from filling the disk.


# Logging Architecture

```text
Application / Service
        |
        v
  writes via:
    syslog API (rsyslog/syslog-ng)    →   /var/log/*.log (text files)
    stdout / stderr (systemd unit)    →   systemd journal (binary, /run/log/journal/)
    direct file write                 →   /var/log/<service>/*.log
        |
        v
journald (systemd-journald) collects:
  - kernel messages (dmesg)
  - systemd service stdout/stderr
  - syslog forwarded messages
        |
        v
journalctl reads binary journal
tail / grep reads text log files
        |
        v
logrotate rotates + compresses old text files (daily/weekly via cron)
```


# Mental Model: Finding a Log

```text
Something broke — where is the log?
        |
        v
Is it a systemd service? (systemctl status <service> shows recent lines)
  → yes: journalctl -u <service> -n 100
        |
        v
Is there a /var/log/<service>/ directory?
  → yes: tail -f /var/log/<service>/error.log
        |
        v
Kernel / hardware issue?
  → dmesg -T | tail -50
  → journalctl -k -b
        |
        v
Auth / SSH / sudo events?
  → /var/log/auth.log (Debian) or /var/log/secure (RHEL)
        |
        v
General system events?
  → /var/log/syslog (Debian) or /var/log/messages (RHEL)
```


# Core Building Blocks

### Log File Locations

| Path | Content | Distro |
|---|---|---|
| `/var/log/syslog` | General system messages | Debian/Ubuntu |
| `/var/log/messages` | General system messages | RHEL/CentOS |
| `/var/log/auth.log` | Login, sudo, SSH events | Debian/Ubuntu |
| `/var/log/secure` | Login, sudo, SSH events | RHEL/CentOS |
| `/var/log/kern.log` | Kernel messages | Debian |
| `/var/log/dmesg` | Boot + kernel ring buffer dump | Both |
| `/var/log/nginx/*.log` | Nginx access + error | Service-specific |
| `/var/log/apache2/*.log` | Apache access + error | Service-specific |

Most log files require `sudo` to read.
- `/var/log/auth.log` (or `secure`) is the first place to check for SSH and sudo issues.

### Following Logs in Real Time

```bash
tail -f /var/log/syslog             # follow new lines (Ctrl+C to stop)
tail -F /var/log/syslog             # like -f but handles log rotation (use -F for services)
tail -f /var/log/nginx/error.log
```
- `tail -F` (capital F) handles log rotation gracefully — prefer over `tail -f` for long-running watches.

### journalctl (systemd journal)

```bash
journalctl                          # all journal entries (oldest first)
journalctl -b                       # current boot only
journalctl -b -1                    # previous boot
journalctl -f                       # follow new entries (like tail -f)
journalctl -n 100                   # last 100 lines
journalctl -p err                   # filter by priority: emerg/alert/crit/err/warning/notice/info/debug
journalctl -u nginx                 # filter by service unit
journalctl -k                       # kernel messages only
journalctl --since "10 minutes ago"
journalctl --since "2025-02-28 10:00" --until "2025-02-28 12:00"
```

Combine flags:

```bash
journalctl -u nginx -f              # follow nginx logs live
journalctl -u nginx -n 50          # last 50 nginx log lines
journalctl -u nginx -p err         # only errors from nginx
journalctl -kf                     # follow kernel messages live
```
- `journalctl -u <service>` is the primary log source for systemd-managed services.
- Priority levels low→high: `debug info notice warning err crit alert emerg` — `err` and above = real problems.
- `journalctl --vacuum-size=500M` trims the journal without stopping the service.

### Time and Timezone

```bash
timedatectl                         # show current time, timezone, NTP status
timedatectl set-timezone Asia/Bangkok
timedatectl set-ntp true            # enable NTP sync
```

- Wrong timezone makes log timestamps misleading and breaks TLS/auth cert validation.
- Keep NTP enabled; all servers in a cluster must be time-synced for log correlation.
- Always check timezone with `timedatectl` when logs seem to be from the wrong time.

### Log Rotation (logrotate)

```bash
cat /etc/logrotate.conf             # global config
ls /etc/logrotate.d/                # per-service configs

# manual run
logrotate -d /etc/logrotate.conf    # dry run (shows what would happen)
logrotate /etc/logrotate.conf       # real run
```

Common options in logrotate config:

| Option | Meaning |
|---|---|
| `daily` / `weekly` / `monthly` | rotation frequency |
| `rotate 4` | keep 4 old files |
| `compress` | gzip old logs |
| `size 100M` | rotate when file reaches 100 MB |
| `missingok` | don't error if log file is missing |
| `notifempty` | don't rotate empty files |
| `postrotate` | run script after rotation (e.g. signal service) |
- `logrotate` runs daily via cron; `postrotate` scripts send reload signals so services reopen log files.

# Troubleshooting Guide

### Service failing — need logs

1. Show last few lines: `systemctl status <service>`.
2. Check full recent log: `journalctl -u <service> -n 100 --no-pager`.

### No logs from service in journal

1. Check if service logs to a file instead: `ls /var/log/<service>/`.

### Disk full — suspected log cause

1. Find largest log files: `du -sh /var/log/* | sort -rh | head`.
2. Check journal size: `journalctl --disk-usage`.
3. Trim journal to 500 MB: `journalctl --vacuum-size=500M`.

### Log timestamps wrong

1. Check timezone and NTP sync status: `timedatectl`.
