# Multi-Factor Authentication (MFA)

- MFA requires two or more independent factors to verify identity, making a single stolen credential insufficient for access.
- Factors fall into three categories: something you know (password), something you have (phone/key), something you are (biometric).
- Phishing-resistant factors (WebAuthn/FIDO2) are the strongest; SMS is the weakest and being deprecated.

# Architecture

```text
Factor Categories:

  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
  │  Knowledge      │  │  Possession     │  │  Inherence      │
  │  (something you │  │  (something you │  │  (something you │
  │   know)         │  │   have)         │  │   are)          │
  │                 │  │                 │  │                 │
  │  - Password     │  │  - Phone (TOTP) │  │  - Fingerprint  │
  │  - PIN          │  │  - Security key │  │  - Face ID      │
  │  - Secret Q&A   │  │  - Smart card   │  │  - Voice        │
  └─────────────────┘  └─────────────────┘  └─────────────────┘

MFA Login Flow:

  User                           Server
    |                              |
    |--- username + password ----->|
    |                              |-- verify password hash
    |                              |
    |<-- "enter 2nd factor" ------|
    |                              |
    |--- TOTP code / key tap ---->|
    |                              |-- verify 2nd factor
    |                              |
    |<-- access granted ----------|
```

# Mental Model

```text
Why MFA works:

  Password only:
    Attacker steals password → full access
    (phishing, breach, keylogger — one factor compromised = game over)

  Password + TOTP:
    Attacker steals password → still blocked
    They also need your phone with the TOTP app
    (two independent factors must be compromised simultaneously)

  Password + WebAuthn:
    Attacker steals password → still blocked
    They also need your physical security key
    AND the key is bound to the domain (phishing doesn't work)
    (strongest combination)

Factor strength ladder:
  SMS (weakest) → TOTP → Push → WebAuthn/FIDO2 (strongest)
```

# Core Building Blocks

### Factor Categories

- **Knowledge** (something you know): password, PIN, security questions. Easiest to steal (phishing, breaches, guessing).
- **Possession** (something you have): phone with TOTP app, hardware security key, smart card. Must be physically stolen.
- **Inherence** (something you are): fingerprint, face recognition, voice. Hard to steal but hard to change if compromised.
- True MFA requires factors from **different categories**. Password + security question = two knowledge factors = NOT true MFA.
- Each factor has a different attack surface — that's why combining them is effective.

Related notes: [004-authentication](./004-authentication.md)

### TOTP (Time-Based One-Time Password)

- Server and client share a secret key (established during enrollment, usually via QR code).
- Every 30 seconds, both sides compute: `HMAC-SHA1(secret, floor(time / 30))` → 6-digit code.
- Client enters the code; server computes the same code and compares.
- RFC 6238; built on top of HOTP (RFC 4226).
- Apps: Google Authenticator, Authy, Microsoft Authenticator, 1Password.
- **Backup codes**: one-time recovery codes generated at enrollment — store securely, they bypass TOTP if phone is lost.
- Weakness: shared secret can be stolen if enrollment QR is intercepted or server's secret store is breached.

```text
Enrollment:
  Server generates secret → QR code → User scans with TOTP app
  Both server and app now share the secret

Verification:
  App: HMAC(secret, time_step) → 847293
  Server: HMAC(secret, time_step) → 847293
  Match → authenticated
```

Related notes: [002-hashing](./002-hashing.md)

### HOTP (HMAC-Based One-Time Password)

- Counter-based variant: `HMAC-SHA1(secret, counter)` → 6-digit code.
- Counter increments after each successful authentication (not time-based).
- Less common than TOTP; used in hardware tokens (e.g., RSA SecurID).
- Problem: if counters get out of sync (user generates codes without using them), authentication fails until resynchronized.

Related notes: [002-hashing](./002-hashing.md)

### U2F / WebAuthn (FIDO2)

- Hardware-based authentication using security keys (YubiKey, Titan Key) or platform authenticators (Touch ID, Windows Hello).
- **How it works**:
  1. Registration: server sends a challenge; key generates a key pair, returns the public key.
  2. Authentication: server sends a challenge; key signs it with the private key; server verifies with stored public key.
- **Phishing-resistant**: the key binds the signature to the exact domain (origin). A phishing site on `evil.com` cannot use a credential registered for `bank.com`.
- **Passkeys**: WebAuthn credentials synced across devices (iCloud Keychain, Google Password Manager). Convenience of passwords + security of public key crypto.
- No shared secret — private key never leaves the hardware device (or secure enclave).
- FIDO2 = WebAuthn (browser API) + CTAP (communication with the key).

```text
Registration:
  Server → challenge → Security Key
  Key generates key pair, signs challenge
  Key → public key + signed response → Server
  Server stores public key

Authentication:
  Server → challenge → Security Key
  Key signs challenge with private key (requires user tap/biometric)
  Key → signed response → Server
  Server verifies with stored public key
```

Related notes: [003-symmetric-vs-asymmetric](./003-symmetric-vs-asymmetric.md)

### Push Notifications

- Server sends a push notification to a registered mobile app; user approves or denies.
- Apps: Duo, Microsoft Authenticator, Okta Verify.
- More convenient than TOTP (tap to approve vs. type a code).
- **Number matching**: server shows a number, user must select the matching number in the app — prevents MFA fatigue attacks.
- Requires internet connectivity on the phone; offline = no push.

Related notes: [004-authentication](./004-authentication.md)

### SMS / Voice

- Server sends a one-time code via text message or voice call to the registered phone number.
- **Weakest MFA factor** — being deprecated by security-conscious organizations:
  - SIM swapping: attacker convinces carrier to transfer your number to their SIM.
  - SS7 attacks: interception of SMS messages on the telecom network.
  - No domain binding: code works regardless of which site requests it (phishable).
- Still better than no MFA at all.
- NIST SP 800-63B recommends against SMS as a primary factor for high-security applications.

Related notes: [004-authentication](./004-authentication.md)

### MFA Attacks and Mitigations

| Attack | How it works | Mitigation |
|--------|-------------|------------|
| SIM swapping | Attacker ports your phone number to their SIM | Use authenticator apps or hardware keys instead of SMS |
| MFA fatigue (push bombing) | Attacker triggers repeated push notifications hoping user approves to stop them | Number matching, rate limiting push requests |
| Real-time phishing proxy | Attacker relays MFA code in real-time between victim and real site | WebAuthn (domain-bound, can't be proxied) |
| Backup code theft | Attacker finds stored backup codes | Encrypt backup codes, store offline |
| TOTP secret theft | Attacker steals the shared secret from server | Hardware keys (no shared secret); encrypt secret store |
| Session hijacking (post-MFA) | Attacker steals session cookie after MFA completes | Short session timeouts, device binding, re-auth for sensitive actions |

Related notes: [009-network-security](./009-network-security.md)

### When to Use Which

| Factor | Security | Convenience | Best for |
|--------|----------|-------------|----------|
| SMS | Low | High | Legacy systems only |
| TOTP | Good | Medium | Default choice for most users |
| Push | Good | High | Enterprise with managed devices |
| WebAuthn | Highest | Medium | High-security accounts, phishing targets |
| Passkeys | High | High | Consumer apps replacing passwords entirely |

- Start with TOTP as the baseline; upgrade to WebAuthn for admin accounts and high-value targets.
- Require MFA for: all admin access, VPN, cloud consoles, email, any system with sensitive data.
- Enforce MFA at the identity provider level (SSO) rather than per-application.

# Troubleshooting Guide

### TOTP codes not working
1. Check clock sync: TOTP requires clocks within 30 seconds. Run `timedatectl` on server; check phone time is set to automatic.
2. Check if correct account: user may have multiple TOTP entries in their app for different environments.
3. Try the previous and next code (server should accept ±1 time step to allow for clock drift).
4. If consistently failing: re-enroll — generate new shared secret and scan new QR code.

### Lost second factor (locked out)
1. Use backup codes if available (generated during enrollment).
2. Admin can temporarily disable MFA on the account for re-enrollment.
3. If using hardware key: use a registered backup key (always register two keys).
4. Identity verification: verify the user through an out-of-band channel before resetting MFA.

### MFA fatigue / push bombing
1. Symptom: user reports receiving many unexpected push notifications.
2. Immediately block the authentication attempts and reset the user's password (attacker has the password).
3. Enable number matching on push notifications (user must enter a number shown on screen).
4. Set rate limits on MFA push requests (e.g., max 3 per minute).
5. Investigate: check auth logs for the source of the login attempts.
