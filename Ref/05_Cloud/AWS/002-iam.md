# IAM (Identity and Access Management)

- IAM controls who (authentication) can do what (authorization) on which AWS resources.
- Principals: Users (people), Roles (services/apps), Groups (collections of users); all are global (not regional).
- Policies are JSON documents that define Allow/Deny rules on specific Actions and Resources.

# Architecture

```text
API Request
    |
    v
[1] Authentication -- Who is calling? (user, role, service)
    |
    v
[2] Request Context -- Action, Resource, Conditions
    |
    v
[3] Policy Evaluation
    |
    +-- Explicit Deny? -----> DENY
    |
    +-- SCP allows? ---------> (if in Organization)
    |
    +-- Resource policy? -----> evaluate
    |
    +-- Permission boundary? -> intersect
    |
    +-- Identity policy? -----> evaluate
    |
    v
[4] Final Decision -- Allow only if no deny AND at least one allow
```

- Every AWS API call passes through this evaluation chain; understanding the order is key to debugging access issues.
- SCPs apply only in AWS Organizations; they restrict what the entire account can do before identity policies are checked.
- Permission boundaries cap the maximum permissions a user/role can receive — they do not grant access on their own.

# Mental Model

```text
Principal (User, Role, Service)
        │
        ▼
Authenticates (password, access key, STS token)
        │
        ▼
Makes API call (e.g. s3:PutObject on arn:aws:s3:::my-bucket/*)
        │
        ▼
IAM evaluates all applicable policies:
  Identity policy (attached to user/role)
  + Resource policy (attached to S3 bucket, KMS key)
  + Permission boundary (if set)
  + SCP (if in AWS Organizations)
        │
        ▼
Explicit Deny wins → then Allow needed → else implicit Deny
```

# Core Building Blocks

### Users and Groups

- **User**: Identity with long-term credentials (password for console, access key for CLI/API).
- **Group**: Collection of users; attach policies to group (not individual users).
- Best practice: use Groups for permission management; avoid attaching policies to individual users.
- IAM is global — users, groups, roles, and policies are not region-specific.

Related notes: [000-core](./000-core.md), [001-aws-overview](./001-aws-overview.md)

### Roles

- **Role**: Identity with temporary credentials; no password or access key.
- **Assume role**: User, service, or external identity assumes a role to get temporary credentials (STS).
- **EC2 instance role**: EC2 assumes role via instance profile; app gets creds from IMDS — no stored keys.
- **Cross-account role**: Allow identity in Account A to assume role in Account B.
- **Service-linked role**: Pre-defined by AWS for specific services (e.g. ECS, RDS).
- Roles provide temporary credentials via STS; users have long-lived credentials.
- Instance profiles are the mechanism for EC2 to assume IAM roles.

Related notes: [000-core](./000-core.md), [004-ec2](./004-ec2.md)

### Policies

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*",
    "Condition": {
      "IpAddress": { "aws:SourceIp": "10.0.0.0/8" }
    }
  }]
}
```

| Element | Description |
|---------|-------------|
| **Effect** | Allow or Deny |
| **Action** | API actions (e.g. `s3:GetObject`, `ec2:*`) |
| **Resource** | ARN(s) the rule applies to |
| **Condition** | Optional: IP, MFA, tag, time, etc. |
- Explicit Deny always wins over any Allow.

Related notes: [000-core](./000-core.md)

### Policy Types

| Type | Attached To | Use Case |
|------|------------|----------|
| Identity policy | User, Group, Role | Standard permissions |
| Resource policy | S3, KMS, SQS, Lambda | Cross-account access, public access |
| Permission boundary | User, Role | Limit maximum permissions |
| SCP | AWS Organization OU | Limit entire account |
- Resource policies (S3 bucket policy, KMS key policy) enable cross-account access.

Related notes: [000-core](./000-core.md), [005-s3](./005-s3.md)

### Best Practices

- **Never use root account** for daily work; enable MFA on root.
- **Use roles** over users; temporary credentials over long-lived keys.
- **Least privilege**: Start with zero permissions, add as needed.
- **Use groups** for permission management.
- **Rotate access keys** if they must exist; prefer roles.
- **Enable CloudTrail** for API audit trail.
- `aws sts get-caller-identity` shows which identity is making the API call.
- Root account should only be used for billing and initial setup; enable MFA immediately.
- CloudTrail logs every API call for audit and forensics.
