# CNI (Container Network Interface)

## Overview
**Why it exists** — Kubernetes needs to assign IPs to pods and wire up their network, but the implementation varies wildly across environments (bare metal, cloud, on-prem). CNI decouples Kubernetes from any specific networking implementation by defining a standard plugin interface.
**What it is** — A specification + library that defines how a runtime (kubelet) calls external programs (plugins) to set up or tear down pod networking. When a pod is created, kubelet calls the configured CNI plugin binary; the plugin allocates an IP, creates the veth pair, and programs routes.
**One-liner** — CNI is the standardized contract that lets Kubernetes say "set up this pod's network" without caring how it's done.

## Architecture (ASCII)

```text
kubelet creates pod
       │
       ▼
kubelet calls CNI plugin binary
  (/opt/cni/bin/<plugin>)
       │
       ├── reads config from /etc/cni/net.d/<config>.conf
       │
       ▼
CNI plugin actions:
  1. Allocate IP from pod CIDR
  2. Create veth pair
  3. Move one veth end into pod netns
  4. Assign IP to pod's eth0
  5. Add default route in pod netns
  6. Connect other veth end to node bridge
  7. Program routes / overlays for cross-node traffic
  8. Optionally enforce NetworkPolicy
       │
       ▼
Pod is ready with IP and working network
```

## Mental Model

CNI is like a plug-in socket for networking. Kubernetes says: "I need this pod networked — here's the namespace, here's the pod info." The CNI plugin handles everything else. You can swap plugins (Flannel → Calico) without changing Kubernetes itself, as long as the plugin follows the CNI spec.

The config files in `/etc/cni/net.d/` are read in lexicographic order — the first one wins. The binary lives in `/opt/cni/bin/`.

## Core Building Blocks

### CNI Configuration File
**Why it exists** — Tells kubelet which plugin to call and how to configure it.
**What it is** — JSON file in `/etc/cni/net.d/` describing the plugin name, IP CIDR, and any plugin-specific settings.
**One-liner** — The recipe file that kubelet hands to the CNI binary when wiring up a pod.

```bash
# View CNI config on a node
ls /etc/cni/net.d/
cat /etc/cni/net.d/10-calico.conflist

# View CNI plugin binaries
ls /opt/cni/bin/
```

### IP Address Management (IPAM)
**Why it exists** — Every pod needs a unique IP from the cluster's pod CIDR; IPAM is the sub-plugin responsible for allocating and releasing those IPs.
**What it is** — A CNI sub-plugin (e.g., `host-local`, `calico-ipam`, `cilium`) that maintains a pool of available IPs and assigns one to each pod.
**One-liner** — The IP address bookkeeper inside the CNI plugin.

```bash
# View pod CIDRs per node
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.podCIDR}{"\n"}{end}'
```

### NetworkPolicy Enforcement
**Why it exists** — By default all pods can talk to all pods; NetworkPolicy allows restricting ingress/egress. Enforcement requires a CNI plugin that supports it.
**What it is** — CNI plugins with policy support (Calico, Cilium, Weave) watch NetworkPolicy objects and program iptables/eBPF rules to allow or deny traffic between pods.
**One-liner** — NetworkPolicy is only enforced if your CNI plugin supports it — Flannel alone does not.

```bash
# Check if NetworkPolicy is being enforced
kubectl get networkpolicy -A

# Test connectivity between pods
kubectl exec <pod-a> -- curl <pod-b-ip>:<port>
```

### Plugin Comparison

| Plugin | Network Model | NetworkPolicy | Performance | Notes |
|--------|--------------|---------------|-------------|-------|
| Calico | BGP or VXLAN overlay | Yes | High | Most popular in production; good for large clusters; flexible routing modes |
| Flannel | VXLAN overlay | No (needs Calico for policy) | Medium | Simple setup; great for learning; lacks built-in policy |
| Weave | Mesh overlay | Yes | Medium | Easy to install; self-healing mesh; lower adoption now |
| Cilium | eBPF (no iptables) | Yes (L3–L7) | Very High | High performance; deep observability; requires Linux 4.9.17+ |

## Troubleshooting

### Pod stuck in ContainerCreating — CNI error
1. Check kubelet logs: `journalctl -u kubelet -n 50 | grep -i cni`.
2. Verify CNI plugin pods are running: `kubectl get pods -n kube-system`.
3. Verify config exists on the node: `ls /etc/cni/net.d/`.
4. Verify binary exists: `ls /opt/cni/bin/`.

### Pod IP not in expected CIDR
1. Check node's pod CIDR: `kubectl get node <node> -o yaml | grep podCIDR`.
2. Check CNI config CIDR matches cluster's `--pod-network-cidr`.
3. Restart CNI plugin pod on the affected node.

### NetworkPolicy not being enforced
1. Confirm your CNI plugin supports NetworkPolicy (see table above).
2. Flannel alone does not enforce policy — install Calico alongside it, or switch CNI.
3. Test with: `kubectl exec <pod> -- curl <blocked-pod-ip>` — should timeout if policy is applied.
